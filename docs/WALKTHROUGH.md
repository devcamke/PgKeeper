# Guided first run: onboard a database and schedule its backups

This walkthrough takes you from nothing to a scheduled, verifiable backup of a
real database using two commands most people meet first: the **`pgkeeper
connect` onboarding wizard** and **`pgkeeper schedule`**. Everything below is
real output from a live run against a Postgres database named
`invoice_development`; substitute your own database wherever it appears.

If you want the exhaustive reference instead, see [USAGE.md](USAGE.md) (every
command) and [SCHEDULING.md](SCHEDULING.md) (scheduling in depth). This page is
the narrative that strings the first few steps together.

## Prerequisites

- `pg_dump` / `pg_restore` / `psql` on your `PATH` (the `postgresql-client`
  package). `pgkeeper doctor` checks this for you.
- A reachable Postgres database and a role that can read it.
- PgKeeper installed (`gem install pgkeeper`, or from a checkout as shown in the
  [README](../README.md#getting-started)).

## Step 1 — the onboarding wizard (`pgkeeper connect`)

`pgkeeper connect` is the guided way in. It collects the connection details,
**live-tests the credentials against the server**, takes a validated backup
schedule (previewing the next few runs), and writes `pgkeeper.yml` — creating a
fresh commented config on a new host, or appending to an existing one without
disturbing its comments or `<%= ENV[...] %>` interpolations.

```sh
pgkeeper connect -c pgkeeper.yml
```

A full session looks like this (the values after each `:` prompt are what you
type — press Enter to accept the `(default)` shown in parentheses):


What just happened:

- **The credential test is a real `psql` round-trip** — the same check
  `pgkeeper doctor` runs — so a typo or an unreachable server is caught up
  front. On a failure the wizard lets you retry (e.g. after starting the
  server), save the config anyway, or abort.
- **The password is never written to the file.** It becomes an env-var
  reference (`<%= ENV["PGKEEPER_INVOICE_DEVELOPMENT_PASSWORD"] %>`); the wizard
  prints the exact variable to export.
- **The schedule is validated and previewed** before it's accepted, so you see
  the next few real run times, not just a cron string you have to trust.

The generated `pgkeeper.yml` is a plain, commented file you own from here on:

```yaml
# PgKeeper configuration — generated by `pgkeeper connect`.
#
# Secrets belong in the environment, not this file: password fields below
# interpolate an env var via ERB. Export it before running pgkeeper.

workdir: "/var/backups/pgkeeper"

# When backups run. Accepts cron ("15 3 * * *"), natural language
# ("every day at 03:15"), or shorthands ("hourly", "daily at 03:15").
schedule: daily at 03:15

compression: gzip

databases:
  - name: invoice_development
    host: 127.0.0.1
    port: 5432
    database: invoice_development
    username: invoice
    password: <%= ENV["PGKEEPER_INVOICE_DEVELOPMENT_PASSWORD"] %>
    format: custom

storage:
  - type: local
    path: "/var/backups/pgkeeper/backups"

retention:
  keep_daily: 7
  keep_weekly: 4
  keep_monthly: 6
```

> **Prefer to skip the wizard?** Copy
> [`config/pgkeeper.example.yml`](../config/pgkeeper.example.yml) to
> `pgkeeper.yml` and edit it by hand — the wizard is a convenience, not a
> requirement.

## Step 2 — set the password, then check and back up

Export the variable the wizard named, then validate the environment and take
the first backup:

```sh
export PGKEEPER_INVOICE_DEVELOPMENT_PASSWORD='…'

pgkeeper validate -c pgkeeper.yml   # config is well-formed
pgkeeper doctor   -c pgkeeper.yml   # tools present, storage + DB reachable
pgkeeper backup   -c pgkeeper.yml   # dump → compress → (encrypt) → store
pgkeeper list     -c pgkeeper.yml   # what landed
pgkeeper verify --deep -c pgkeeper.yml   # restore into a throwaway DB to prove it
```

`verify --deep` is the one worth building a habit around: it restores the
backup into a scratch database and checks it, turning "we have backups" into
"we have backups that provably restore." See [RESTORE.md](RESTORE.md) for the
restore runbook.

## Step 3 — schedule the jobs

A `schedule:` in the config is a *plan*; it doesn't run anything until you wire
it into something that fires on time. `pgkeeper schedule` reads the config and
emits that wiring for you. First, see the resolved plan:

```sh
pgkeeper schedule print -c pgkeeper.yml
```

```
backup all: daily at 03:15 (cron: 15 3 * * *) (all databases)
```

### Schedule maintenance too, not just the backup

A backup you never verify or prune drifts. Add a `maintenance:` block so
**verify** and **prune** are scheduled as first-class jobs alongside the
backup:

```yaml
maintenance:
  verify:
    schedule: weekly on sunday at 04:00
    deep: true            # Tier-3: restore into a throwaway scratch database
  prune:
    schedule: daily at 05:00
    apply: true           # actually delete (omit/false = dry-run only)
```

Now the plan covers all three jobs:

```
backup all: daily at 03:15 (cron: 15 3 * * *) (all databases)
verify all: weekly on sunday at 04:00 (cron: 0 4 * * 0) --deep (all databases)
prune all: daily at 05:00 (cron: 0 5 * * *) --apply (all databases)
```

Pick **one** of the three ways to run that plan:

### Option A — cron

`pgkeeper schedule install` prints ready-to-use crontab lines, each guarded by
its own `flock` so a slow run never overlaps the next tick:

```sh
pgkeeper schedule install -c pgkeeper.yml
```

```crontab
# pgkeeper managed
15 3 * * * /usr/bin/flock -n /var/backups/pgkeeper/.cron-all.lock pgkeeper backup --config pgkeeper.yml >> /var/backups/pgkeeper/pgkeeper.log 2>&1
0 4 * * 0 /usr/bin/flock -n /var/backups/pgkeeper/.cron-verify-all.lock pgkeeper verify --config pgkeeper.yml --deep >> /var/backups/pgkeeper/pgkeeper.log 2>&1
0 5 * * * /usr/bin/flock -n /var/backups/pgkeeper/.cron-prune-all.lock pgkeeper prune --config pgkeeper.yml --apply >> /var/backups/pgkeeper/pgkeeper.log 2>&1
```

Pipe them into your crontab (review first):

```sh
pgkeeper schedule install -c pgkeeper.yml | crontab -
```

### Option B — systemd timers

`--systemd` emits a `.service` + `.timer` pair per job, with `Persistent=true`
(catch up a missed run after downtime) and a `RandomizedDelaySec` stagger:

```sh
pgkeeper schedule install --systemd --jitter 300 --output /etc/systemd/system -c pgkeeper.yml
systemctl daemon-reload
systemctl enable --now pgkeeper-backup-all.timer pgkeeper-verify-all.timer pgkeeper-prune-all.timer
```

Each pair looks like:

```ini
# ===== pgkeeper-backup-all.service =====
[Unit]
Description=PgKeeper backup (all)
After=network-online.target
Wants=network-online.target

[Service]
Type=oneshot
ExecStart=pgkeeper backup --config pgkeeper.yml

# ===== pgkeeper-backup-all.timer =====
[Unit]
Description=PgKeeper backup timer (all)

[Timer]
OnCalendar=*-*-* 03:15:00
Persistent=true
RandomizedDelaySec=300

[Install]
WantedBy=timers.target
```

### Option C — the built-in daemon (containers)

No cron or systemd? `pgkeeper daemon` runs every schedule in-process, with
optional jitter — ideal for a container:

```sh
pgkeeper daemon --jitter 300 -c pgkeeper.yml
```

This is the process the Docker image runs by default; see the
[README](../README.md#docker) for the compose setup.
