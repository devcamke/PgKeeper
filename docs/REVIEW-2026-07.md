# PgKeeper — Full Application Review (July 2026)

_A code-grounded review of the entire application at commit `0ec0142` (post-PITR
Stage 7): every module under `lib/` was read in full, the test suite and RuboCop
were run, CI/packaging were inspected, and every finding below was verified
against the actual code — the highest-severity ones re-verified independently a
second time. This complements [ASSESSMENT.md](ASSESSMENT.md) (a product-fit
assessment) with a defect-focused engineering review._

**Baseline health:** the suite passes (464 runs, 1,314 assertions, 0 failures,
17 skips — the skips are live-Postgres integration tests, which CI *does* run
against a real Postgres 16 service). RuboCop: 136 files, no offenses. CI covers
lint, unit + integration tests, a Ruby 3.2/3.3/3.4 compat matrix, weekly
`bundler-audit`, dependency review, and gem-build sanity.

---

## Verdict

The logical-dump core remains production-grade: the crypto, subprocess
discipline, retention rails, storage contract, and web auth are genuinely
well built (see [Strengths](#strengths)). But this review found **one critical
and roughly a dozen high-severity defects**, concentrated in three places:

1. **The PITR subsystem is not yet trustworthy end-to-end.** The documented
   `archive_command` integration wedges permanently on the first
   `pg_basebackup` (C1); several restore/verify paths can silently lose data
   or green-light an unrecoverable chain (H5–H9). PITR shipped in stages and
   it shows: the pieces work in isolation, but the seams between Postgres's
   real behavior and PgKeeper's model have gaps.
2. **Failure reporting fails open in a few load-bearing spots.** A run that
   stored *zero* copies of a backup reports "partial" (H1); a plain-format
   restore literally cannot fail (H2); the destructive-restore guard skips
   itself on probe error (H3).
3. **The daemon/container lifecycle is fragile.** A DST fall-back hour crashes
   the daemon (H4); the container cannot be stopped gracefully and can bake a
   stale config into the image (H10–H11).

None of this undermines the architecture — nearly every fix below is local and
small. But the critical and high items should be fixed before the next release,
and before PITR is marketed as dependable.

## Critical

### C1. `archive_command` wedges permanently on backup/history files
`lib/pgkeeper/pitr/wal_archiver.rb:26,41` + `lib/pgkeeper/cli.rb:88-91`

`WalArchiver#archive_file` raises for any name not matching
`/\A[0-9A-F]{24}\z/` — the comment says the exclusion of backup-label and
history files is deliberate. That is correct for spool draining, but this same
method backs the documented server config
`archive_command = 'pgkeeper wal archive-file --cluster N %p %f'`, and
PostgreSQL invokes `archive_command` for `.backup` and `.history` files too.
The first `pg_basebackup` (or any promote) hands the wrapper a name like
`000000010000000000000005.00000028.backup`; pgkeeper exits 1; Postgres retries
that same file forever because archiving is strictly ordered. From that moment
**no further WAL is archived**: `pg_wal` grows until the disk fills, and the
recovery window silently stops advancing. The WAL-lag dead-man's switch will
eventually fire, but the primary integration path is broken by design.

**Fix:** accept (archive, or explicitly acknowledge-and-succeed) `*.backup` and
`*.history` names in `archive_file`; archive `.history` files (see H8).

## High

### H1. A run whose every upload fails reports `:partial`, not `:failure`
`lib/pgkeeper/orchestrator.rb:211-216`

`derive_status` returns `:partial` whenever any destination failed — including
when **all** of them failed. The staging directory (the only copy of the dump)
is then deleted in `backup_database`'s `ensure`. With a single configured
destination (the common case) and a disk-full/S3 outage, zero copies of the
backup exist anywhere, yet the exit code, history row, and notification all say
"partial". Monitoring keyed on failure misses a run that stored nothing.
No test covers the all-destinations-failed case.

### H2. Plain-format restores cannot fail
`lib/pgkeeper/restore.rb:107`

`run_restore` invokes `psql ... -v ON_ERROR_STOP=0 -f dump.sql`. With
`ON_ERROR_STOP` off, psql exits 0 even when statements error, so a truncated or
mid-file-failing plain dump restores partially and PgKeeper prints
"✓ Restore complete". The project's own verifier documents this exact trap and
uses `ON_ERROR_STOP=1` (`verify.rb:145-153`); the real restore path contradicts
it. The Restorer has no unit tests, and the integration tests only exercise
custom format.

### H3. The destructive-restore guard fails open
`lib/pgkeeper/restore.rb:88-101`

`guard_target!` does `return unless status.success?` — if the table-count probe
fails for any reason other than "query ran" (transient connection error, auth
hiccup), the `--force` check is silently skipped and the restore proceeds
against a possibly non-empty database. A failed guard probe should abort.

### H4. The daemon crashes during the DST fall-back hour
`lib/pgkeeper/daemon.rb:31,62`

`entry.schedule.next_time(from: ...)` raises `ArgumentError` when `from` falls
in the ambiguous repeated hour (verified experimentally against the bundled
fugit 1.13.0 / et-orbi 1.4.0: `Cannot determine timezone from "EDT"`). The
rescue in `run_entry` covers only the runner call; the rescheduling call at
`daemon.rb:62` and the initial computation at `:31` are unguarded, so the
exception exits `Daemon#run` and the process dies — on fall-back night, with an
hourly schedule, deterministically. This is the "silently dead scheduler"
failure the daemon exists to prevent. (The dashboard guards this same call —
`web/dashboard.rb:110-113` — the daemon does not.)

Related, medium: on **spring-forward** day a "daily at 02:30" schedule is
skipped entirely (fugit returns the *next* day's 02:30 rather than shifting to
03:00 as vixie cron does) — one silently missed backup per year.

### H5. PITR `restore_command` bakes an unresolvable relative config path
`lib/pgkeeper/pitr/restore.rb:144`

`write_recovery` emits
`restore_command = 'pgkeeper wal fetch ... --config #{@config.source}'` using
the config path exactly as given. Postgres runs `restore_command` with the data
directory as CWD, so a relative path (`pgkeeper.yml`, the default discovery
path) never resolves: every fetch exits 1, Postgres treats that as
segment-not-found, and with `--to latest` (no recovery target,
`recovery_target_action=promote`) it **promotes at the end of the base
backup's embedded WAL — silently discarding all archived WAL after the base.**
With explicit targets it at least fails loudly. Fix: `File.expand_path` the
config source when generating the command.

### H6. WAL chain verification misses a gap at the head of the chain
`lib/pgkeeper/pitr/verify.rb:35-38,55-63`

`verify --pitr` checks contiguity only *between* archived segments; nothing
checks that the chain starts at the base's `start_segment`. If archiving failed
for the first segment(s) after the base, verify reports "recovery chain
intact" (and prints a range starting at the base's segment) while a real
restore stops dead at the end of the base backup. Tests cover a mid-chain gap
but not a head gap. Related, medium: verification anchors at the *newest* base,
so the chain from the retention-window anchor base — the one that makes the
7-day promise — is never verified at all.

### H7. Point-in-time base selection uses start, not the consistency point
`lib/pgkeeper/pitr/restore.rb:60-71`

`pick` chooses the newest base whose *start* time/LSN ≤ target, but recovery
can only stop at points **after** the backup's consistency point (its stop
LSN) — [PITR-DESIGN.md](PITR-DESIGN.md) §8 says stop_lsn ≤ target. A target
that falls inside a base backup's start–stop window picks that base and
Postgres refuses ("requested recovery stop point is before consistent recovery
point") even though the previous base could serve it. `BaseBackup` doesn't
record `stop_lsn`/`finished_at` on the artifact, so the picker has nothing to
compare against — that's the root fix. The same start-vs-finish confusion in
`pitr/retention.rb:39-47` means a long base backup crossing the window edge can
leave the edge of the promised recovery window unrecoverable. `--to-name`
targets are worse: they always pick the newest base (`sorted.last`), which may
postdate the named restore point entirely, with no fallback to an older base.

### H8. Timeline history files are never archived — post-failover recovery silently stops at the old timeline
`lib/pgkeeper/pitr/wal_archiver.rb:26,56`

`NNNNNNNN.history` files are rejected by `archive_file` (see C1) and skipped by
`archive_spool`'s glob. After a promote, timeline-2 segments are archived but
`00000002.history` never is; a later restore (default
`recovery_target_timeline=latest`) asks for the history file, gets exit 1,
concludes the latest timeline is 1, and replays only pre-failover WAL —
everything after the failover is unreachable even though its WAL is in the
archive. `verify --pitr` explicitly scopes timeline switches out, so nothing
catches it.

### H9. Plain `pgkeeper verify` goes permanently red once PITR artifacts exist
`lib/pgkeeper/verify.rb:55-64` vs `lib/pgkeeper/pruner.rb:43-45`

PITR base/WAL manifests carry `database: <cluster name>`, so
`Catalog#databases` includes clusters. The pruner explicitly subtracts cluster
names; the verifier does not. With PITR enabled, `pgkeeper verify` (including
the scheduled `maintenance: verify` job) selects the cluster's newest artifact
— a WAL segment — and tier-2 runs `pg_restore --list` on it, which fails every
time. Scheduled verification and its alerting become permanently red, and
`list` shows every WAL segment as an unverified backup set.

### H10. Docker: a local `pgkeeper.yml` is baked into the image and shadows the mounted config
`Dockerfile:32`, `.dockerignore`, `lib/pgkeeper/cli.rb:13`

`docker-compose.example.yml` tells the operator to create `pgkeeper.yml` in the
build-context directory; `.dockerignore` does not exclude it; `COPY . .` puts
it at `/app/pgkeeper.yml` — the **first** entry of `DEFAULT_CONFIG_PATHS`
(relative to `WORKDIR /app`), searched before the mounted
`/etc/pgkeeper/pgkeeper.yml`. Host-side config edits silently do nothing until
the image is rebuilt, and any inlined secrets are baked into image layers.
Fix: add `pgkeeper.yml`/`config/pgkeeper.yml` to `.dockerignore` (and consider
having the container default to `-c /etc/pgkeeper/pgkeeper.yml`).

### H11. Containers cannot be stopped gracefully; a dead dashboard goes unnoticed
`lib/pgkeeper/daemon.rb` (no signal traps), `docker/entrypoint.sh:17-28`

The daemon installs no signal handlers. As Docker PID 1 (default CMD path,
which `exec`s), unhandled SIGTERM is ignored, so every `docker stop` waits out
the grace period and SIGKILLs — potentially mid-`pg_dump`/mid-upload. In
`daemon-with-web`, `sh` stays PID 1 without `exec`; POSIX shells defer traps
until the foreground command exits, so TERM is never forwarded to the daemon
either. And despite the comment "If either process dies, take the container
down", nothing waits on `$web_pid` — if the dashboard dies (auth missing, port
conflict) the container keeps running with it silently dead. Fix: trap
INT/TERM in the daemon for a clean loop exit; in the entrypoint, run both
children and `wait -n` on either, forwarding signals.

### H12. `pitr.base_backup.schedule` is validated, documented — and never scheduled
`lib/pgkeeper/pitr/config.rb:102,147-171` vs `lib/pgkeeper/scheduler.rb:46-49`

The Scheduler builds only backup/verify/prune entries; nothing anywhere
constructs scheduled base backups (only the CLI does, manually). An operator
who configures `base_backup: {schedule: "daily at 02:00"}` and runs
`pgkeeper schedule install` gets no recurring bases, and no alarm compensates:
`Pitr::Health` has no "newest base too old" check (`stalled?` is WAL-only, and
`window_short?` actually *improves* as the oldest base ages). Status stays
green while the only base drifts months stale.

## Medium

- **Zip-slip in `decompress_tree`** — `lib/pgkeeper/compress/zip.rb:62-76`
  joins `entry.name` onto the destination with no containment check, and
  rubyzip 2.4.1 skips its own safety check when handed an explicit path. A
  tampered directory-format artifact (the ransomware/bucket-write threat model
  SECURITY.md itself discusses — feasible for unencrypted artifacts since the
  manifest sidecar can be recomputed) writes files outside the restore dir as
  the restore user. Add a `File.expand_path` containment check per entry.
- **Globals dump failure voids an already-uploaded primary** —
  `lib/pgkeeper/orchestrator.rb:218-226`: if `pg_dumpall` fails after the
  primary artifact uploaded everywhere, the run reports `:failure` with zero
  artifacts and the uploads become orphans invisible to history, anomaly
  baselines, and `total_bytes` (the pruner does still see the files).
- **Verification can vacuously pass** — `lib/pgkeeper/verify.rb:92-93,107-110`:
  tier 1 skips artifacts whose manifest lacks a checksum; tier 2 for plain
  format checks only that the first 4 KiB are non-empty, so a 90%-truncated
  plain dump passes non-deep verify, gets stamped `verified_at`, and then
  *anchors the prune-protection rail* while older good backups age out.
- **The verified-protection rail exists only on the primary destination** —
  `verify.rb:50-52` stamps manifests on the local-preferred destination only;
  on every other destination `verified?` is always false, so the "never prune
  the last verified backup or newer" rail (`pruner.rb:10-12`) doesn't apply to
  cloud copies.
- **Catalog swallows all manifest read errors** — `catalog.rb:96-104` rescues
  `StandardError → nil`, so a transient download failure silently drops a
  backup from the catalog for that run: `protected_after` and the
  "never delete newest" rail then anchor on the wrong set. Separately, a
  valid-JSON manifest with an unparseable `started_at` yields
  `timestamp: nil`, and `backup_sets`' sort crashes `list`/`verify`/`prune`
  for the whole destination (PITR paths crash the same way:
  `pitr/restore.rb:61`, `pitr/verify.rb:66`, `pruner.rb:91`).
- **Removing/renaming a PITR cluster reclassifies its artifacts as prunable
  logical dumps** — `pruner.rb:44-45` subtracts *current* config cluster
  names; artifacts of a deleted cluster fall into the GFS policy, which
  deletes WAL mid-chain and destroys point-in-time recoverability of what it
  keeps.
- **`prune` can delete PITR artifacts while printing "No retention policy
  configured; nothing to prune."** — `cli.rb:526-533` keys the message on the
  logical policy only, suppressing the PITR dry-run preview in
  PITR-only-retention configs.
- **PITR restore drops tablespaces silently** — `pitr/restore.rb:116-121`
  extracts only `base.tar`; per-tablespace `<oid>.tar` files bundled by
  `pg_basebackup -Ft` are ignored with no error. Also `--force` doesn't clear
  a non-empty data dir, and `postgresql.auto.conf` is *appended*, so repeated
  attempts stack conflicting `restore_command`/target blocks. And no
  checksum/`pg_verifybackup` check runs on fetched WAL or the restored base
  (design §8 step 4), so bit-rot surfaces only as an opaque replay error.
- **`pitr.destinations` is parsed, validated, documented — and consumed by
  nothing** (`pitr/config.rb:101,137-145`): base backups and WAL always fan
  out to every destination.
- **Web/scheduler serialization claim is false for verify/prune** — the UI and
  docs say "Every action runs through the same lock", but only backup and base
  backup take `.pgkeeper.lock`; a web-triggered `prune --apply` can run
  against a half-uploaded set mid-backup, and unbounded per-POST threads
  (`web/jobs.rb:35-39`) let an authenticated caller stack concurrent deep
  verifies.
- **`gpg` and `zstd` shell-outs bypass the Subprocess deadline design**
  (`crypto/gpg.rb:68`, `compress/zstd.rb:40`): bare `Open3.capture3`, no
  timeout — a wedged gpg-agent or stalled mount hangs the run forever, the
  exact failure `Subprocess` exists to prevent. (Latent: `Subprocess.capture3`
  itself writes stdin before spawning readers/arming the deadline —
  `subprocess.rb:36-44` — a backpressure deadlock no lib caller currently
  triggers.)
- **Doctor blesses hosts that cannot back up** — `doctor.rb:112-118`: an older
  `pg_dump` vs newer server is only `:warn` and `healthy?` ignores warnings,
  so doctor exits 0 while every scheduled dump will hard-fail (pg_dump refuses
  newer-major servers — the Dockerfile's own comment says so). Same pattern
  for a configured destination whose cloud SDK gem is missing (`doctor.rb:88-89`).
  Doctor has zero tests.
- **Wizard fallback can silently drop existing databases** —
  `config_writer.rb:63-66` only matches block-style `databases:`; a flow-style
  list fails the regex, a *second* `databases:` key is appended at EOF, and
  Psych's last-key-wins removes the originals from the effective config while
  they still appear in the file.
- **Config errors other than `Psych::SyntaxError` escape as raw backtraces** —
  `config.rb:90-95`: ERB `KeyError`/`NameError` and `Psych::DisallowedClass`
  (e.g. an unquoted date) bypass `ConfigError`, so the CLI exits 1 (the
  documented "partial" code) with a backtrace instead of the clean report.
- **OAuth-token adapters never refresh** — Dropbox/SharePoint/Google adapters
  cache the token for the adapter's lifetime with no expiry tracking or
  401-retry; any single run longer than the token lifetime (~1 h Graph, ~4 h
  Dropbox) dies mid-upload, non-retryably.
- **Image/runtime gaps** — the shipped image runs Ruby 4.0, which no CI job
  tests (compat matrix stops at 3.4) and the image is never built in CI;
  `Gemfile.lock` is `.dockerignore`d so image gem versions are unpinned;
  the container runs as root with no `HEALTHCHECK`; `release.yml` publishes on
  tag without running tests.

## Low (notable)

- `derive_status`-adjacent: artifact names have 1-second resolution and
  `Local` finalizes with an unconditional rename — two runs of a small DB in
  the same second silently overwrite the earlier artifact
  (`orchestrator.rb:221`, `storage/local.rb:41`).
- DB slug permits `.`/`..` (`config.rb:737-739`), letting a hostile config
  write/delete outside the storage root; config-author controlled.
- S3 + Object Lock: versioned-bucket deletes write delete markers, so pruned
  locked objects persist invisibly and nothing ever cleans noncurrent versions;
  `object_lock.retain_days` missing coerces to 0 days — configured lock,
  zero protection, silently (`storage/s3.rb:92-108`).
- Unauthenticated `/readyz` leaks the workdir absolute path and raw SQLite
  error text (`web/health.rb:44-57`).
- Web downloads: tempfile leaked when a destination download fails
  (`web/app.rb:212-215`); Content-Disposition relies on Puma to neutralize
  CRLF in storage-controlled names (`web/app.rb:222,271`).
- Basic-auth comparison short-circuits on username mismatch, leaking username
  validity via timing; each comparison is itself constant-time (`web/auth.rb:84-85`).
- Idle clusters trip the WAL dead-man's switch (no `archive_timeout` → no
  segments → "lag" grows; `pitr/health.rb:92-93`).
- `say_error` writes to stdout (`cli.rb:702-704`); several read-only commands
  lack `rescue Error` and backtrace with exit 1, colliding with the
  "1 = partial" exit-code contract.
- `ask_secret` silently falls back to echoed password entry if `noecho`
  raises (`prompt.rb:91-101`); logfmt values don't escape newlines
  (`logging.rb:72-77`); systemd unit generation lacks quoting/`%%` escaping
  (`scheduler/systemd.rb:24,31`); `ConfigWriter.write` resets a 0600 config
  to umask default (`config_writer.rb:26-33`); anomaly `shrink_pct: 100`
  silently disables shrink detection (`anomaly.rb:75`); zip tree packaging
  omits dotfiles (`compress/zip.rb:47`) — harmless for pg_dump output, but the
  same method packages base-backup directories.

## Strengths (verified, not repeated from marketing)

- **Crypto is textbook**: fresh random 96-bit IV per artifact, tag verified
  before success, PBKDF2-HMAC-SHA256 at 210k iterations with per-artifact
  salt, keyring rotation, guaranteed cleanup of partial plaintext.
- **Subprocess discipline**: argv-array exec (no shell anywhere), children in
  their own process group, TERM→grace→KILL escalation — correct for pg_dump's
  helper forks.
- **Storage layer**: a shared contract test suite keeps every backend honest;
  centralized retry-with-jittered-backoff; post-upload size verification;
  Local finalizes copy → chmod 0600 → fsync → rename.
- **Retention logic is pure and fails toward keeping data**, both GFS
  (`retention.rb`) and PITR (`pitr/retention.rb`); prune defaults to dry-run;
  PITR artifacts are firewalled from the logical GFS policy.
- **flock handling avoids the classic unlink/reopen race** and is tested with
  real forked-process contention.
- **Web security fundamentals**: catalog-allowlisted downloads (traversal
  explicitly tested), Bearer-scheme CSRF for the JSON API, notifier isolation
  (delivery failure can never fail a backup), correct Prometheus label
  escaping and atomic textfile writes.
- **Config validation** accumulates all problems, rejects unknown keys at
  every level, handles the YAML `on:`→`true` footgun, and `ConfigWriter` edits
  by targeted text surgery so comments and ERB survive.
- **Testability as a design property**: injectable clock/sleeper/runner/IO
  throughout; WAL segment arithmetic (LSN mapping, 0xFF rollover) is correct
  and tested.

## Biggest test-coverage gaps

1. `Verifier` and `Restorer` have **no unit tests** (only env-gated
   integration tests, custom-format only) — which is exactly where H2/H3 live.
2. `Doctor` has zero tests.
3. No test for all-destinations-failed status (H1), head-of-chain WAL gaps
   (H6), `.backup`/`.history` names hitting `archive_file` (C1/H8), targets
   inside a base's start–stop window (H7), verify with PITR artifacts present
   (H9), or zip-slip/tamper on `decompress_tree`.
4. The Docker image is never built (let alone smoke-tested) in CI, and the
   Ruby the image ships (4.0) is outside the compat matrix.

## Suggested fix order

1. **C1 + H8** (archive_command accepts `.backup`/`.history`; archive history
   files) — unblocks PITR's primary integration path.
2. **H2 + H3** (`ON_ERROR_STOP=1`; guard fails closed) — three-line fixes on
   the most dangerous command.
3. **H1** (all-failed → `:failure`) and **H4** (rescue/normalize `next_time`;
   add signal traps) — restore trust in failure reporting and the daemon.
4. **H5, H9, H10** — one-line-ish fixes (expand config path; subtract cluster
   names in Verifier like Pruner; extend `.dockerignore`).
5. **H6, H7, H12** and the PITR mediums — require recording `stop_lsn`/
   `finished_at` on artifacts and threading them through pick/retention/verify;
   schedule base backups or fail validation on an unschedulable cadence.
6. Then the medium tier, roughly in listed order.
